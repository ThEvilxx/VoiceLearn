# VoiceLearn PDF 提取能力分析

> 分析日期：2026-05-31
> 当前状态：用于讨论，非 bug 修复计划

## 当前工具链

PDF 提取使用 `pypdf.PdfReader` ——零布局感知的纯文本提取器。

`requirements.txt` 中已安装但**未使用**的替代方案：`pdfplumber`（支持版面区域感知、表格提取）。

---

## 问题清单

### 1. 版面布局

| 子问题 | 当前状态 | pypdf 行为 | pdfplumber 改善 |
|--------|---------|-----------|----------------|
| 双栏文字交叉 | ❌ 未解决 | 按 PDF 内容流物理顺序读取，左栏片段与右栏片段交替出现 | **部分解决** — 支持按页面区域（bbox）分离左右栏 |
| 跨栏图表/表格 | ❌ 未解决 | 无区域概念，图表内碎片文本插入正文中间 | **部分解决** — 可过滤或隔离图表区 |
| 脚注 | ❌ 未解决 | 引用标号后紧跟页面底部脚注文字 | **不能** |
| 页眉页脚 | ❌ 未解决 | 会议名、页码、作者名穿插在跨页段落中间 | **部分解决** — 可裁剪页面顶部/底部固定区域 |

**根因：** PDF 内容流是物理流，不是逻辑流。pypdf 不知道哪里有栏、哪里是正文区、哪里是页脚。

### 2. 数学公式

| 状态 | pypdf 行为 |
|------|-----------|
| ❌ 无解 | PDF 中的公式被 LaTeX 渲染为**矢量图形指令**（线条、曲线、特定字体字形的位置坐标），不是 Unicode 字符。`\frac{a^2}{b_1}` 在 PDF 文本层根本不存在 |

提取出来的结果可能是一串随机空格、缺字符、或者无意义的碎片符号。上标被拆成独立小号字符、分子和分母分别位于分数线上下方、求和符号 ∑ 被渲染为字体字形没有对应 Unicode 码点。

**根因：PDF 是渲染格式，不是结构化文档。** 公式的数学含义在渲染阶段已被丢弃。唯一的正确路径是直接使用 LaTeX 源码（arXiv 论文提供 `.tex` 下载）或通过 OCR+数学识别模型重建，两者都不在当前系统范围内。

### 3. 字体编码玄学

| 子问题 | 当前状态 | pypdf 行为 |
|--------|---------|-----------|
| 连字陷阱（fi→□, define→de ne） | ❌ 未解决 | 依赖 PDF 内置 ToUnicode CMap。该表写得好的论文 OK（如 arXiv），写得烂或自定义字体的论文丢失字符 |
| 自建编码表（Hello→!@#$X） | ❌ 未解决 | 作者用自定义字体且 ToUnicode 损坏时，pypdf 只能输出原始字符码，显示为乱码 |

**根因：** 学术 PDF 字体生态极其碎片化。arXiv 的自动编译管线产出的 PDF 质量高，但某些会议论文经过多层排版工具链（Word→PDF、InDesign→PDF）后字体映射表质量不可控。

### 4. 代码块缩进

| 状态 | pypdf 行为 |
|------|-----------|
| ❌ 无解 | PDF 不保留空白字符的前导位置信息。所有前置空格和 Tab 被吞掉，代码全部顶格输出 |

**根因：** 代码的精髓在缩进，而 pypdf 按"有效字符"提取，不保留排版间距。这跟 PDF 的内部表示有关——缩进通常通过移动文本起始的 x 坐标实现，而不是插入空格字符。

## 为什么之前测试没暴露这些问题

上线前的测试集是四篇 arXiv 论文（Transformer、CoT、InstructGPT、附带 LLaMA），它们全为**单栏排版**、有**标准 ToUnicode CMap**（arXiv 自动编译管线生成）、且测试以对话检索为目的（我们问"注意力机制是什么"——即使有少数字符失真，RAG 仍然能匹配到正确 chunk）。

这意味着系统在"理想输入"下通过了全链路验证，但以下类型论文可能出问题：
- 双栏排版的会议论文（IEEE、CVPR 标准模板）
- 包含大量 inline 公式的纯理论证明论文
- 低质量扫描档转 PDF（ToUnicode 表损坏率高）
- AI/ML 论文中常见的伪代码页

## 改进方向（v1.x 优先序）

| 优先级 | 改动 | 行数 | 效果 |
|--------|------|------|------|
| P0 | `_load_pdf()` 改为 pdfplumber 实现 | ~20 行 | 双栏分离、页眉页脚过滤、表格提取 |
| P1 | 支持 `.tex` 文件上传 | ~15 行 loader | 数学公式精准保留——LaTeX 源码处理公式和代码是完美的 |
| P2 | 连字 fallback 字典 | ~10 行 loader | 常见连字手动映射（fi→fi, fl→fl, ffi→ffi, Æ→AE） |
| P3 | 代码块缩进保留 | 无法在 PDF 层解决 | 需配合 `.tex` 源码或单独上传代码文件 |
| P3 | 数学公式重建 | 无法在 PDF 层解决 | 需引入 MathPix / Nougat 等专用工具或换 OCR 通道 |

## 边界声明

本文档坦承当前系统的 PDF 局限。核心用户场景（arXiv 英文论文→语音问答）的测试已充分验证，双栏/公式密集论文可能需要用户承担部分提取质量损失。这不是代码 bug，而是 `pypdf` 作为通用 PDF 库的能力边界。
