# AI Bootstrap

`ai-bootstrap` 的默认目标不是从零生成一套新规范，而是帮助一个已有项目做 AI 协作规范体检，并补齐缺口。

优先处理已有宪法项目：

1. 先读已有规范入口。
2. 按七层输出诊断报告。
3. 等用户确认。
4. 再生成 `*.draft` 草稿。

只有项目没有任何宪法或规范入口时，才进入新项目初始化。

---

## 工作原则

1. **诊断优先**：先给状态和证据，不直接写文件。
2. **补齐优先**：已有宪法时只生成 supplement，不生成替代性完整宪法。
3. **全面优先**：覆盖宪法、规范、上下文、工具、护栏、验证、技能七层，不在单点做过深推断。
4. **不污染项目**：过程记录默认写到临时目录；项目内只写用户确认过的 `*.draft`。
5. **不覆盖现有文件**：所有项目内产物必须带 `.draft` 后缀。
6. **少模板化**：证据不足时写“待确认问题”，不要伪造命令、目录职责或团队流程。
7. **中文优先**：生成给用户阅读的宪法、规则、技能、报告和待确认问题默认使用中文；代码标识符、命令、文件名、协议字段保留英文。

---

## 阶段 0：目标确认

先判断项目属于哪种状态：

| 状态 | 判断方式 | 默认动作 |
|------|----------|----------|
| 已有宪法 / 规范 | 存在 `AGENTS.md`、项目规则目录、贡献说明、团队规范等入口 | 诊断 + 补齐 |
| 无明显规范 | 没有项目级 AI 协作入口，也缺少规则 / SOP | 新项目初始化 |
| 只想评估 | 用户明确要求只看缺口、不写文件 | 只输出诊断报告 |

如果能自行判断，就不要打断用户提问。只有目标模式确实不清楚时才问。

再判断当前工具的真实配置承载。默认只支持 `cursor`、`codex`、`claude-code` 三类目标；其他工具必须先由执行者按项目事实判断。

| 工具 / 场景 | 默认承载 | 生成策略 |
|------------|----------|----------|
| Cursor | `.cursor/rules`、`.cursor/skills`、`.cursor/hooks.json` | 可生成对应 `.draft`，但仍需用户确认后激活 |
| Codex | `AGENTS.md`、`.codex/rules`、`.codex/skills/rules-check` | 生成规则草稿，并生成 `rules-check` skill 草稿，用于要求 Codex 任务前读取规则 |
| Claude Code | `CLAUDE.md`、`AGENTS.md` 或项目已有 Claude Code 配置 | 不固定 rules / skills 子目录；由执行者判断真实承载，无法确认时只生成草稿和待确认问题 |

## 阶段 1：事实采集

优先使用辅助脚本采集基础事实：

```bash
python3 "${AI_SKILL_ROOT}/scripts/scan_project.py" "$(pwd)" --config-dir <实际配置目录>
```

`<实际配置目录>` 必须由当前执行的 AI 先判断后填入，例如 `.codex`、`.cursor` 或项目约定的其他目录。不要把占位符原样传给脚本，也不要省略 `--config-dir` 让脚本猜。

脚本只负责采集事实，不能替代你的诊断。至少还要人工阅读：

- 项目宪法或主规范入口。
- README / CONTRIBUTING / docs 入口。
- Makefile / package scripts / CI workflow。
- 现有 rules / skills / hooks / ignore 文件。

过程记录默认写到临时目录，例如：

```text
/tmp/ai-bootstrap-<project>-<timestamp>/
├── profile.json
├── decisions.md
└── generated-files.md
```

除非用户明确要求，不要在项目内创建 `.ai-bootstrap/`、`bootstrap-profile.json`、`bootstrap-summary.md` 等过程文件。

## 阶段 2：七层诊断

面向用户输出七层报告。每层先给简单状态，再给详细描述和优化建议。

状态只能使用：

- `已具备`：已有内容能指导 AI 稳定执行。
- `缺失`：没有发现对应入口。
- `需补齐`：已有入口，但缺少关键事实、命令、边界或验证。
- `冲突`：多个位置表达不一致，必须先让用户确认事实源。

报告格式：

| 层级 | 状态 | 详细诊断 | 优化建议 |
|------|------|----------|----------|
| 宪法 | 已具备 / 缺失 / 需补齐 / 冲突 | 是否有项目级总入口，是否说明事实源、边界、禁令、交付标准 | 生成 supplement / 补事实源 / 补交付门禁 |
| 规范 | 已具备 / 缺失 / 需补齐 / 冲突 | 是否有编码、Git、API、安全、文档等细分规则 | 拆分规则、删除重复、补缺失领域 |
| 上下文 | 已具备 / 缺失 / 需补齐 / 冲突 | AI 是否知道该先读哪些 README、设计文档、目录和示例 | 补上下文路由、减少口头依赖 |
| 工具 | 已具备 / 缺失 / 需补齐 / 冲突 | 是否有真实可执行命令、脚本、Makefile、CI 入口 | 补命令说明、校准不存在的命令 |
| 护栏 | 已具备 / 缺失 / 需补齐 / 冲突 | 是否有 AI 外部的自动拦截、审批、CI 或权限控制 | 把纯文本禁令升级为 hook / CI / 权限策略 |
| 验证 | 已具备 / 缺失 / 需补齐 / 冲突 | 是否说明修改后跑什么测试、构建、检查 | 为不同变更类型补验证矩阵 |
| 技能 | 已具备 / 缺失 / 需补齐 / 冲突 | 是否把复杂高频流程沉淀为 skill / SOP | 只为多步骤、易错、有前置条件的流程生成候选 skill |

诊断报告后追加：

1. **优先级建议**：最多 5 条，按影响从高到低排序。
2. **待确认问题**：只列真正影响草稿生成的问题。
3. **建议生成物**：说明准备生成哪些 `*.draft`，等待用户确认。

用户确认前，不写项目文件。

## 阶段 3：生成草稿

### 已有宪法项目

默认只生成补齐内容：

- `AGENTS.md.supplement.draft`
- 缺失的规则草稿：仅在当前工具确认有项目规则目录时生成，例如 Cursor 的 `<CONFIG_DIR>/rules/<name>.<ext>.draft` 或 Codex 的 `.codex/rules/<name>.md.draft`
- 候选 skill 草稿或候选清单：仅在当前工具确认有项目 skill 目录时生成，例如 Cursor 的 `<CONFIG_DIR>/skills/<name>/SKILL.md.draft`；Codex 默认生成 `.codex/skills/rules-check/SKILL.md.draft`
- 必要时生成 `待确认问题.md.draft`

`AGENTS.md.supplement.draft` 只写建议新增或修正的段落，不复制已有宪法全文。

### 新项目

只有在没有宪法或主规范入口时，才生成：

- `AGENTS.md.draft`
- 工具确认支持项目规则目录时，才生成 rules 草稿
- 工具确认支持项目 hook 配置时，才生成 hook 示例
- 工具确认支持项目 skill 目录时，才生成候选 skill 草稿

可以参考模板结构，但内容必须来自项目事实。证据不足时写“待确认问题”，不要留大段泛化待补充内容。

### 规则草稿

辅助脚本可用于生成基础规则草稿：

```bash
python3 "${AI_SKILL_ROOT}/scripts/generate_rules.py" \
  "$(pwd)" --preset <PRESET> --config-dir <实际配置目录>
```

执行前必须把 `<实际配置目录>` 替换为真实目录。Codex 项目通常填 `.codex`，Cursor 项目通常填 `.cursor`；如果项目已有明确规则目录，以项目事实为准。
如果当前工具没有本工具包固定的项目规则目录，脚本会拒绝生成。`--config-dir .codex` 会生成 `.codex/rules/*.md.draft`，并额外生成 `.codex/skills/rules-check/SKILL.md.draft`，让 Codex 在任务前显式读取 `AGENTS.md` 和 `.codex/rules`。

脚本输出后必须人工检查并删改空泛内容。规则文件必须：

- 有可触发的 frontmatter。
- 正文不超过 100 行。
- 写清适用范围、执行步骤和验证方式。
- 跳过已有 live 文件，只生成 `.draft`。
- 正文默认使用中文，只有命令、路径、代码标识符和协议字段保留英文。

### Skill 候选

不要把普通命令包装成 skill。只为满足以下条件的流程生成候选：

- 多步骤，顺序重要。
- 新人或 AI 容易搞错。
- 涉及环境变量、外部依赖、权限、发布、数据迁移或跨系统协作。
- 有明确完成标准和验证方式。

默认先输出候选清单，用户确认后再生成 `SKILL.md.draft`。

## 护栏判断标准

不要把 `AGENTS.md`、规则文件、skill 流程或 checklist 称为护栏。它们只是告诉 AI 应该怎么做。

真正的护栏必须满足：

1. 在 AI 之外执行。
2. 拦在具体动作上，例如写文件、执行命令、提交、推送、调用远程写接口。
3. 能自动产生强制结果，例如拒绝、要求审批、自动格式化、阻断提交、阻断 CI。

推荐优先识别这些护栏候选：

- 写入前拦截：禁止改运行时配置、禁止改生成文件、禁止写疑似 secret。
- 命令审批：破坏性命令、外部写操作、远程系统变更必须经 permission gate。
- 服务端门禁：protected branch、required checks、pre-receive hook、RBAC、只读 token。
- 写后补救：格式化、drift check、CI fail-fast。

如果只能写成“AI 应该记住”，不要叫护栏；把它放进规则或流程。

## 阶段 4：交付与激活

展示生成物清单：

```text
✅ Bootstrap 诊断完成

七层状态：
  宪法：<状态>
  规范：<状态>
  上下文：<状态>
  工具：<状态>
  护栏：<状态>
  验证：<状态>
  技能：<状态>

已生成草稿：
  AGENTS.md.supplement.draft / AGENTS.md.draft
  <工具真实规则目录>/*.<ext>.draft（如适用）
  <工具真实技能目录>/*/SKILL.md.draft（如适用）

过程记录：
  <临时目录路径>
```

用户确认后，再激活 draft：

```bash
bash <skill_install_dir>/ai-bootstrap/scripts/activate_drafts.sh "$(pwd)"
```

脚本只把 `*.draft` 重命名为正式文件；已有同名正式文件会跳过，不覆盖。

激活后提示用户继续执行 `ai-config-check`。

---

## 注意事项

- 默认用户的项目已经有宪法或基础规范；不要急着生成完整新宪法。
- 默认过程记录不写入项目，用完即可丢弃。
- 所有项目内产物用 `*.draft` 后缀。
- 诊断报告必须按七层输出。
- 分发包是一次性产物，不是长期事实源。
- 分析要具体，不要泛泛而谈。
